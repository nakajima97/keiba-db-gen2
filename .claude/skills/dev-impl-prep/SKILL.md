## 概要

AI駆動開発ワークフロー（`docs/workflow/ai-dev-workflow.md`）のステップ3「実装準備」を実行する。
要件定義フェーズで確定したアウトプット（API仕様・UI仕様・DBスキーマ等）を守るためのテストを、
まず箇条書きで計画し、人間の承認を得てから実装することがゴール。

## 引数

- `<issue番号>`（必須）: 実装準備を行うGitHub issueの番号
- 未指定の場合: 「issue番号を指定してください。例: /dev-impl-prep 42」と伝えて終了する

## エージェント構成

```
メインスキル（オーケストレーター）
├── Step1: Issue確認（メイン）
├── Step2: コードベース調査 → Explore エージェント
│             ↓ 調査サマリーを受け取る
├── Step3: テスト計画（箇条書き）（メイン）
├── Step4: 人間確認・合意（メイン）
└── Step5: テスト実装 → general-purpose エージェント
```

## 手順

### Step 1: Issue確認

1. `gh issue view <番号> --comments` でissueの内容とコメントを取得する
   - 要件定義フェーズで記録された確定仕様（コメント）を必ず確認する
2. `docs/specs/` 配下の関連ファイルを読み、確定済みアウトプットを把握する
   - 対象例: `api-list.md`, `openapi.yaml`, `screen-list.md`, `er-diagram.md`
3. テスト対象となるアウトプットを整理する（APIエンドポイント・UIコンポーネント・DBスキーマ等）

### Step 2: コードベース調査（Explore エージェント）

以下を調査させ、調査サマリーをメインに返させる:

- 既存テストのディレクトリ構造とファイル命名規則
- 使用しているテストフレームワーク（フロントはVitest、バックエンドは任意）
- 既存テストファイルの書き方（describe/it/expect の構成パターン等）
- テストファイルの保存場所の規則（`__tests__` ディレクトリ、`.test.ts` サフィックス等）

Exploreエージェントへの指示例:
```
以下を調査してサマリーを返してください:
1. 既存テストファイルのディレクトリ構造と命名規則
2. 使用しているテストフレームワーク（package.json, requirements.txt等から判断）
3. 既存テストファイルの書き方のパターン（describe/it/test の構成）
4. テストファイルの保存場所の規則
```

### Step 3: テスト計画（箇条書き）（メイン）

Step1・Step2の情報をもとに、テストケースを箇条書きで列挙する。**コードは書かない。**

以下のカテゴリでテストケースを整理する:

**APIテスト**（APIエンドポイントの追加・変更がある場合）:
- 各エンドポイントごとに正常系・異常系・バリデーションのテストケースを列挙

**UIコンポーネントテスト**（画面の追加・変更がある場合）:
- 各コンポーネントごとにレンダリング・props・インタラクションのテストケースを列挙

**その他**（DB変更がある場合など）:
- スキーマ検証・マイグレーション検証等のテストケースを列挙

### Step 4: 人間への確認・フィードバック反映（メイン）

1. テスト計画（箇条書き）をユーザーに提示する
2. フィードバックを受けて修正する（複数回対応）
3. ユーザーが承認したら、承認済みテスト計画を `gh issue comment` でissueに記録する
4. Step5へ進む

### Step 5: テスト実装（general-purpose エージェント）

Step4で承認されたテスト計画と、Step2のコードベース調査サマリーをコンテキストとして渡し、
テストコードを実装させる。

general-purposeエージェントへの指示例:
```
以下の調査サマリーと承認済みテスト計画をもとに、テストコードを実装してください。
[調査サマリー]
[承認済みテスト計画]

- ビジネスロジックは実装しない（テストコードのみ）
- 既存テストのパターン・命名規則に従う
- 実装したテストファイルの一覧を返してください
```

完了後:
1. 実装したテストファイルの一覧をユーザーに提示する
2. `gh issue comment` で実装完了をissueに記録する
3. 「実装準備フェーズが完了しました」と伝えてフェーズ終了

## 注意事項

- `gh` コマンドが使えない場合はエラーをそのままユーザーに伝える
- テスト計画（箇条書き）は人間が承認するまでテストコードを書かない
- 既存テストのパターン・命名規則に必ず従う
- issueに記録するコメントは日本語で書く
- 変更が不要なカテゴリのテストは計画に含めない（APIの変更がなければAPIテストは不要）
