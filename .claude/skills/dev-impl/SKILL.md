## 概要

AI駆動開発ワークフロー（`docs/workflow/ai-dev-workflow.md`）のステップ4「実装」を実行する。
ステップ2（要件定義）で確定した仕様とステップ3（実装準備）で合意したテスト計画を入力として、
補助テストの作成・実装・セルフレビュー・自動修正を経て、人間が動作確認できる状態にすることがゴール。

## 引数

- `<issue番号>`（必須）: 実装を行うGitHub issueの番号
- 未指定の場合: 「issue番号を指定してください。例: /dev-impl 42」と伝えて終了する

## エージェント構成

```
メインスキル（オーケストレーター）
├── Step1: Issue確認・仕様把握（メイン）
├── Step2: 補助テスト作成（メイン）
├── Step3: 実装 → general-purpose エージェント
│             ↓ 実装ファイル一覧を受け取る
├── Step4: セルフレビュー・自動修正ループ（メイン、最大2回）
│   ├── 試行1: 問題検出 → general-purpose エージェントで修正
│   └── 試行2（問題が残る場合）: 再修正 → 解消しなければ人間に報告して終了
└── Step5: 人間による確認・記録（メイン）
```

## 手順

### Step 1: Issue確認・仕様把握

1. `gh issue view <番号> --comments` でissueの内容とすべてのコメントを取得する
   - ステップ2（要件定義フェーズ）で記録された確定仕様コメントを特定する
   - ステップ3（実装準備フェーズ）で記録された承認済みテスト計画コメントを特定する
   - これらのコメントが見当たらない場合は前フェーズが未完了の可能性があるため、その旨をユーザーに伝えて終了する
2. `docs/specs/` 配下の関連ファイルを読み、確定済みアウトプットを把握する
   - 対象例: `api-list.md`, `openapi.yaml`, `screen-list.md`, `er-diagram.md`
3. `docs/architecture/` のアーキテクチャ定義を読む（ディレクトリ構造・命名規則）
4. 以下の情報をStep2〜4のコンテキストとして保持する:
   - 確定仕様のサマリー（対象のAPIエンドポイント・UIコンポーネント・DBスキーマ）
   - 承認済みテスト計画の箇条書き（ステップ3の成果物）
   - アーキテクチャ定義（ディレクトリ構造・命名規則）

### Step 2: 補助テスト作成（4-1）

ステップ3で作成した仕様守護テストに加え、疎結合な設計を保つための補助テストを追加作成する。
**この補助テストは実装コードを書く前に作成する。**

補助テストの目的と対象:
- コンポーネント間の依存関係がアーキテクチャ境界を越えていないことを確認するテスト
- 外部依存（API・DB・外部サービス）をモック化してユニットテストできることを確認するテスト
- ステップ3のテストと重複しない範囲で追加する

作成手順:
1. ステップ3の承認済みテスト計画を確認し、まだカバーされていない疎結合設計の検証ポイントを洗い出す
2. 補助テストファイルを作成する（既存のテストパターン・命名規則に従う）
3. `gh issue comment` で補助テスト作成完了をissueに記録する:
   ```
   ## 4-1: 補助テスト作成完了
   作成したテストファイル:
   - `<ファイルパス1>`
   - `<ファイルパス2>`
   ```

### Step 3: 実装（4-2）— general-purpose エージェント

Step1で整理した確定仕様・承認済みテスト計画・アーキテクチャ定義をコンテキストとして渡し、
実際のコードを実装させる。

general-purposeエージェントへの指示例:
```
以下の確定仕様・テスト計画・アーキテクチャ定義をもとに、コードを実装してください。

[確定仕様サマリー]
[承認済みテスト計画（ステップ3の成果物）]
[補助テスト一覧（Step2の成果物）]
[アーキテクチャ定義（docs/architecture/ の内容）]

実装の方針:
- docs/architecture/ のディレクトリ構造・命名規則に必ず従う
- ステップ3で作成済みのテストと補助テストをパスするように実装する
- APIエンドポイントはopenapi.yamlまたはapi-list.mdで確定した仕様に正確に従う
- UIコンポーネントはscreen-list.md・screen-transition.mdで確定した仕様に正確に従う
- 既存コードのパターンに従って実装する

完了後に以下を返してください:
- 作成・変更したファイルの一覧
- 実装の概要（何を実装したか）
- パスできなかったテストがあれば一覧と失敗理由
```

完了後:
1. 実装ファイル一覧と概要をメインに受け取る
2. `gh issue comment` で実装完了をissueに記録する:
   ```
   ## 4-2: 実装完了
   作成・変更したファイル:
   - `<ファイルパス1>`（新規作成）
   - `<ファイルパス2>`（変更）
   実装概要: <何を実装したかの説明>
   ```

### Step 4: セルフレビュー・自動修正ループ（4-3）

実装コードに対して品質チェックを行い、問題があれば人間の承認なしに自動修正する。
2回の試行で修正できなかった場合は人間に報告して終了する。

#### レビューの観点

以下の順序でチェックする:

**1. テスト結果の確認**
- テストスイートがすべてパスしているか
  - テストランナーのコマンドは `docs/architecture/` または `package.json`/`README.md` から判断する
- 実装準備フェーズ（ステップ3）でissueに記録された承認済みテスト計画の各項目が実装されており、壊れていないか
  - `gh issue view <番号> --comments` で承認済みテスト計画コメントを取得して確認する

**2. 仕様準拠の確認**
- `docs/specs/` の確定仕様（openapi.yaml・api-list.md・screen-list.md等）と実装が一致しているか

**3. コード品質の確認**
- 既存コードの命名規則・パターンに従っているか
- 不要な複雑さ・重複コードがないか
- アーキテクチャ境界を越えた依存関係がないか（例: UIコンポーネントから直接DBアクセスするなど）

#### 試行1

1. 上記の観点でレビューを実施する
2. 問題が見つかった場合: 問題リストを作成し、general-purposeエージェントに修正させる
   ```
   以下の問題を修正してください:
   [問題リスト]

   - 機能の変更は行わない（修正のみ）
   - 修正後のファイル一覧を返してください
   ```
3. 問題がなかった場合: Step5へ進む

#### 試行2（試行1で問題が残った場合）

1. 修正後のコードを再度同じ観点でレビューする
2. 問題が残っている場合: general-purposeエージェントに再修正させる
3. 再修正後のコードを最終確認する
   - 問題が解消された場合: Step5へ進む
   - 問題が残っている場合: **自動修正をここで打ち切り、人間に報告する**

#### 2回で修正できなかった場合の報告

`gh issue comment` でissueに記録し、ユーザーに以下を伝えて終了する:

```
## 4-3: セルフレビュー — 自動修正できませんでした

2回の修正試行で以下の問題が解消できませんでした:

### 未解決の問題
- <問題1の詳細>
- <問題2の詳細>

### 試行1の修正内容
- <試行1で変更したファイルと変更内容>

### 試行2の修正内容
- <試行2で変更したファイルと変更内容>

手動での確認・修正をお願いします。
```

#### セルフレビュー完了時の記録

問題がなかった場合・修正で解消できた場合、`gh issue comment` でissueに記録する:
```
## 4-3: セルフレビュー完了
レビュー結果: <問題なし / X件の問題を修正>
修正内容: <修正した内容の概要（修正が発生した場合のみ）>
```

### Step 5: 人間による確認（4-4）

1. 実装の結果をユーザーに提示する:
   - 作成・変更したファイルの一覧（補助テスト・実装ファイル・修正ファイルをまとめて）
   - セルフレビューの結果（問題なし or 修正した内容）
   - 動作確認のために必要な手順（起動方法・テスト実行コマンド等）

2. 以下の形式でユーザーに伝える:
   ```
   ## 実装フェーズが完了しました

   ### 作成・変更したファイル
   - `<ファイルパス>`（補助テスト）
   - `<ファイルパス>`（実装）

   ### セルフレビュー結果
   <結果の概要>

   ### 動作確認のお願い
   以下の手順で動作を確認してください:
   1. <手順1>
   2. <手順2>

   確認完了後、issueをクローズしてください。
   ```

3. `gh issue comment` で実装フェーズ完了をissueに記録する:
   ```
   ## 4-4: 実装フェーズ完了 — 人間による確認待ち

   すべての工程が完了しました。動作確認をお願いします。

   ### 成果物サマリー
   - 補助テスト: <ファイル一覧>
   - 実装ファイル: <ファイル一覧>
   - セルフレビュー: <結果>
   ```
4. 作成・変更したファイルをコミットする（`git add` で対象ファイルを個別に追加し、`git add .` は使わない）
5. 次のステップを案内する:
   ```
   動作確認が完了したら、PRを作成してください:
   /create-pr
   ```

## 注意事項

- `gh` コマンドが使えない場合はエラーをそのままユーザーに伝える
- ステップ2（要件定義）・ステップ3（実装準備）のissueコメントが見当たらない場合は、
  その旨をユーザーに伝えてフェーズを終了する（前のフェーズが未完了の可能性がある）
- `docs/architecture/` のアーキテクチャ定義を必ず参照し、既存コードの命名規則・ディレクトリ構造に従う
- セルフレビューでの自動修正は最大2回まで。3回目は行わず必ず人間に報告する
- 補助テストは疎結合設計の検証を目的とする。ステップ3のテストと重複する内容は書かない
- issueに記録するコメントは日本語で書く
- 実装エージェントに渡す指示には確定仕様・テスト計画・アーキテクチャ定義を必ずすべて含める
