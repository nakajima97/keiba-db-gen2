## 概要

AI駆動開発ワークフロー（`docs/workflow/ai-dev-workflow.md`）のステップ2「要件定義」を実行する。
アウトプット駆動開発の考え方に基づき、具体的な成果物を通じて人間との合意形成を取ることがゴール。

## 引数

- `<issue番号>`（必須）: 要件定義を行うGitHub issueの番号
- 未指定の場合: 「issue番号を指定してください。例: /dev-requirements 42」と伝えて終了する

## エージェント構成

```
メインスキル（オーケストレーター）
├── Step1: Issue確認（メイン）
├── Step2: コードベース調査 → Explore エージェント
│             ↓ 調査サマリーを受け取る
├── Step3: アウトプット生成 → general-purpose エージェント
│             ↓ 生成ファイル一覧を受け取る
├── Step4: ドキュメント更新（メイン）
└── Step5: 人間確認・フィードバック（メイン）
```

## 手順

### Step 1: Issue確認

1. `gh issue view <番号>` でissue内容を取得する
2. 機能要件・制約の不明点を洗い出す
3. 不明点があれば **1回にまとめて** ユーザーに確認する（複数回に分けない）
4. ユーザーから回答を得たら、確認内容と回答を `gh issue comment` でissueに記録する
   - 不明点がなかった場合はこの記録をスキップする

### Step 2: コードベース調査（Explore エージェント）

以下を調査させ、調査サマリーをメインに返させる:

- `docs/specs/` 配下の現状（存在する場合 — 画面一覧・ER図等の既存内容を把握）
- `docs/architecture/` のアーキテクチャ定義（ディレクトリ構造・命名規則・使用技術）
- issueに関連する既存コードのパターン・ディレクトリ構造
- 使用しているバックエンドフレームワーク（FastAPI/Laravelなど — Step3でのOpenAPI生成方針の判断に使う）

Exploreエージェントへの指示例:
```
以下を調査してサマリーを返してください:
1. docs/specs/ の現状（存在すれば各ファイルの内容）
2. docs/architecture/ のアーキテクチャ定義
3. issue内容「<issueタイトルと概要>」に関連する既存コード
4. バックエンドフレームワーク（package.json, requirements.txt, composer.json等から判断）
```

### Step 3: アウトプット生成（general-purpose エージェント）

Step2の調査サマリーをコンテキストとして渡し、以下を即時ファイル保存させる。
人間が視覚的・具体的に確認して合意できる成果物を生成することがゴール。
実装コード（処理ロジック）は書かない。

| 成果物 | 条件 | 内容・保存先 |
|---|---|---|
| 静的画面 | 画面の追加・変更がある場合 | presentationalコンポーネントのみの変更 → Storyファイル（モックデータのみ）<br>シンプルなページ追加（条件分岐が少ない） → ページコンポーネントに静的UIを直接実装（ロジックなし）<br>複数の条件状態を持つフォーム・複雑なUI → presentationalコンポーネントに切り出してStoryファイルで各状態を表現 |
| OpenAPI定義 | APIエンドポイントの追加・変更がある場合 | `docs/specs/openapi.yaml`（リクエスト・レスポンスのスキーマ） |
| DBスキーマ | DB構造の変更がある場合 | マイグレーションファイル（カラム定義のみ） |
| ログ定義 | 新規ログ出力がある場合 | `docs/specs/log-spec.md`（ログ形式・出力条件） |
| 非同期メッセージ定義 | イベント・キューが発生する場合 | `docs/specs/events.md`（イベント・メッセージスキーマ） |

general-purposeエージェントへの指示例:
```
以下の調査サマリーとissue内容をもとに、アウトプットを生成してください。
[調査サマリー]
[issue内容]

- 実装コード（処理ロジック）は書かない
- 静的画面は変更の性質で生成方法を選ぶ:
  - presentationalのみ → Story
  - シンプルなページ追加 → ページに直接実装
  - 複数の条件状態を持つフォーム・複雑なUI → presentationalコンポーネント + Story
- 既存のディレクトリ構造・命名規則に従う
- 生成したファイルの一覧を返してください
```

### Step 4: ドキュメント更新（メイン）

以下を即時ファイル保存する。変更がない項目はスキップ。
`docs/specs/` のファイルが存在しない場合は `docs/template/` からコピーして新規作成する。

| ファイル | 内容 | 更新条件 |
|---|---|---|
| `docs/specs/screen-list.md` | 新規画面を行として追記 | 画面の追加・変更がある場合 |
| `docs/specs/screen-transition.md` | 遷移をmermaid flowchartに追記 | 画面遷移の変更がある場合 |
| `docs/specs/er-diagram.md` | テーブル・カラムをmermaid erDiagramに追記・修正 | DB構造の変更がある場合 |
| `docs/specs/api-list.md` | 新規エンドポイントを行として追記（概要一覧） | REST APIエンドポイント（JSONレスポンスを返すもの）の追加・変更がある場合（画面を返すルートは対象外） |
| `docs/specs/openapi.yaml` | Step3で生成済み（追加操作不要） | Step3でOpenAPI YAMLを生成した場合 |

> `api-list.md` は REST API（JSONレスポンスを返すエンドポイント）のみを記載する。`openapi.yaml` はその機械可読の完全仕様。画面ルートは `screen-list.md` に記載し `api-list.md` には含めない。

### Step 5: 人間への確認・フィードバック反映（メイン）

1. 生成・更新したファイルの一覧と変更概要をユーザーに提示する
2. フィードバックを受けて修正する（複数回対応）
3. ユーザーが確定したら、以下を `gh issue comment` でissueに記録する:
   - フィードバック内容（何をどう変えたか）
   - 確定した仕様のサマリー（どのファイルが確定したか）
4. 生成・更新したファイルをコミットする（`git add` で対象ファイルを個別に追加し、`git add .` は使わない）
5. 「要件定義フェーズが完了しました」と伝え、次のコマンドを案内する:
   ```
   次のフェーズへ進む場合は以下を実行してください:
   /dev-impl-prep <issue番号>
   ```

## 注意事項

- `gh` コマンドが使えない場合はエラーをそのままユーザーに伝える
- アーキテクチャ定義（`docs/architecture/`）を必ず参照し、既存コードの命名規則・ディレクトリ構造に従う
- 変更が不要なドキュメントは更新しない（空振りの更新をしない）
- issueに記録するコメントは日本語で書く
- `api-list.md` に記載するのは REST API（JSON レスポンスを返すエンドポイント）のみ。画面を返すルートは REST API ではないため `screen-list.md` に記載し `api-list.md` には含めない。
